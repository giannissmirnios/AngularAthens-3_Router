What is Angular Router and how it works?


Angular Router is an official Angular routing library, written and maintained by the Angular Core Team. It’s a JavaScript router implementation that’s designed to work with Angular and is packaged as @angular/router.

    Angular Router takes care of the duties of a JavaScript router:

  • Activates all required Angular components to compose a page when a user navigates to a certain URL
  • Lets users navigate from one page to another without page reload
  • It updates the browser’s history so the user can use the back and forward buttons when navigating back and forth between pages.

    • Additionally

  • Redirect a URL to another URL
  • Resolve data before a page is displayed
  • Lazy load parts of our application.

Router Resolvers

    General Routing Flow

  • User clicks the link
  • Angular loads the respective component

    • Routing Flow with Resolver

  • User clicks the link
  • Angular executes certain code and returns a value or observable
  • You can collect the returned value or observable  in constructor or in ngOnInit, in the component which is about to load
  • Use the collected the data for your purpose
  • Now you can load your component

How to use a resolver

When to use a resolver

    Whenever you need to get the data before the component initialization

  • This means that the data shown is an integral part of the component; thus the component has no reason to exist (be shown) without its fully loaded data.
    • Showing a prefilled form
    • Showing the user’s booking
    • Showing a map – a partially loaded map is still useful to the user
    • Showing the details of a place on the map

When to use a resolver

    Single Responsibility

  • A components only responsibility is showing the data and managing the data interaction with the user
  • It should not:
    • manage the loading state of a request
    • route to another page if the data is not there
    • handle multiple services to collect all the data required

When NOT to use a resolver

  • A resolver leads to the data being retrieved synchronously – which can be a good pattern or an anti-pattern depending on the component’s role and functionality
  • Can’t fine tune the applications loading spinners down to component level
  • None of the component’s content is displayed until the data is fetched

Scaling an Angular application using multiple router modules

  • Each functional page should have it’s own bookmarkable/refreshable URL
  • After ~20 pages/URLS the router module tends to become difficult to maintain
  • Should be divided in separate router modules

Tree structure to be maintainable

  • Folders and their modules should have the exact same name with their expected URL
  • A clear strategy should be kept regarding data sharing and components/services sharing
    • Redux, ngrx/store
    • Shared components/services/etc. folder
    • Shared library module (?)

To lazy-load or not

  • While scaling an architectural decision should be made to include or not lazy loading
  • This decision can happen in a module per module basis
    • What % of users is expected to reach each module? How often?
    • Do we care about server load?
    • Is the initial load time less important than having fast access to all routes (for ex. betting app)
    • Is the application intended for mobile use? Do we want to save some mobile data from its users?
